The Node.js backend (`backend-node/`) is the central API gateway for the Kairo platform. It handles user authentication, business logic, session management, and acts as the orchestrator between the frontend, the PostgreSQL database, and the Python AI microservice.

## Middleware stack

Middleware is applied in the following order in `src/server.js`:

1. **CORS** â€” enforces the allowed-origins list derived from `FRONTEND_URL`
2. **`OPTIONS` preflight handler** â€” responds to all `OPTIONS /{*path}` requests with CORS headers
3. **Morgan** â€” logs HTTP requests in `dev` format
4. **JSON body parser** â€” parses `application/json` bodies up to 5 MB
5. **URL-encoded body parser** â€” parses form submissions up to 5 MB
6. **Session** â€” cookie-based sessions backed by PostgreSQL (`connect-pg-simple`)
7. **Passport initialize** â€” sets up the Passport authentication framework
8. **Passport session** â€” restores authenticated user from the session store

## CORS configuration

The CORS behavior differs between development and production:

| Environment | Allowed origins |
|-------------|----------------|
| Development (`NODE_ENV != production`) | `http://localhost:5500`, `http://localhost:5173` |
| Production | The origin extracted from `FRONTEND_URL` only |

The `FRONTEND_URL` value is parsed with `new URL()` to extract only the origin component, so including a path in the URL is safe. If `FRONTEND_URL` is not set in production, the server logs a warning and temporarily allows all origins.

Credentials (`credentials: true`) are always enabled, which is required for cookie-based session authentication.

## Rate limiting

Two rate limiters are configured:

| Limiter | Window | Max requests | Applied to |
|---------|--------|--------------|------------|
| `authLimiter` | 15 minutes | 20 per IP | Authentication routes (`/api/auth`) |
| `otpLimiter` | 10 minutes | 5 per IP | OTP verification endpoints |

Rate limit responses use the `RateLimit-*` standard headers and return:

```json
{ "error": "Too many requests, please try again later." }
```

## Session configuration

Sessions are stored in a PostgreSQL table named `session`, created automatically by `connect-pg-simple` if it does not exist.

| Setting | Value |
|---------|-------|
| Cookie name | `riwi.sid` |
| Max age | 24 hours |
| `httpOnly` | `true` (not accessible from JavaScript) |
| `secure` | `true` in production, `false` in development |
| `sameSite` | `none` in production, `lax` in development |
| Store | PostgreSQL via `DATABASE_URL` or `DATABASE_DIRECT_URL` |

<Info>
  `sameSite: 'none'` with `secure: true` is required for cross-origin cookie delivery in production, such as when the frontend is on GitHub Pages and the API is on Railway.
</Info>

## Database connection

The pool is configured in `src/config/database.js` with the following settings:

| Setting | Value |
|---------|-------|
| Max connections | 10 |
| Idle timeout | 30 seconds |
| Connection timeout | 30 seconds |
| Query timeout | 30 seconds |
| SSL | Enabled for Supabase; disable with `DB_SSL=false` for local Postgres |

At startup, `testConnection()` runs a `SELECT NOW()` handshake query. If this fails, the server exits with code 1.

## OAuth configuration

GitHub and Google OAuth strategies are registered in `src/config/passport.js`. The OAuth callback base URL is determined automatically:

- **Production**: `https://kairo-integrative-project-turing-production.up.railway.app`
- **Development**: `http://localhost:{PORT}`

OAuth callback URLs:
- GitHub: `{BASE_URL}/api/auth/github/callback`
- Google: `{BASE_URL}/api/auth/google/callback`

Register these exact URLs as authorized redirect URIs in your GitHub and Google OAuth application settings.

## Production deployment notes

<Warning>
  Set `NODE_ENV=production` in your production environment. This enables secure cookies, restricts CORS to `FRONTEND_URL` only, and suppresses verbose error messages in API responses.
</Warning>

- The server binds to `0.0.0.0` (all interfaces), which is required for containerized deployments.
- `app.set('trust proxy', 1)` is set, which is necessary when running behind a reverse proxy or load balancer (Railway, Heroku, etc.) so that rate limiting uses the real client IP.
- Session cookies require `secure: true` and `sameSite: 'none'` for cross-origin setups. Both are automatically applied when `NODE_ENV=production`.
- Use the `start` script (`node src/server.js`) rather than `nodemon` in production.
