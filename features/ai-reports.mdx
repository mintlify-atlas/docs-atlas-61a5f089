---
title: "AI-Generated Reports"
description: "How Kairo uses the Groq LLM to produce executive performance reports for Team Leaders, including clan summaries, risk assessments, and action items."
---

Kairo can generate AI-written performance reports for Team Leaders on demand. A report analyzes a clan's current data and produces a structured executive summary with a risk assessment, identified concerns, and concrete recommendations.

Reports are stored in the `ai_reports` table and tracked in the `ai_generation_log` audit table. All report generation endpoints require role `tl`.

## Report target types

The `report_target_enum` type defines the scope of a report:

| Target | Description |
|---|---|
| `clan` | Covers all coders in a specific clan. This is the type generated by the `/generate-report` endpoint. |
| `coder` | Covers a single coder's individual performance. |
| `cohort` | Covers an entire cohort across all clans. |

## Generating a report — `POST /api/ia/generate-report`

This is the Node.js proxy endpoint. It requires an active session with role `tl` and forwards the request payload to the Python service.

**Required role:** `tl`

**Request body:**

<ParamField body="clan" type="string" required>
  The clan identifier (e.g., `"Turing"`, `"Tesla"`, `"McCarthy"`).
</ParamField>

<ParamField body="tl_id" type="number" required>
  The Team Leader's user ID.
</ParamField>

<ParamField body="total_coders" type="number" required>
  Total number of coders in the clan.
</ParamField>

<ParamField body="average_score" type="number" required>
  The clan's current Moodle average score (0–100).
</ParamField>

<ParamField body="high_risk_count" type="number" required>
  Number of coders with an active `high` or `critical` risk flag.
</ParamField>

<ParamField body="top_struggling_topics" type="array">
  List of topic names where coders are struggling most. Optional.
</ParamField>

<ParamField body="soft_skills_summary" type="object">
  Key-value map of soft skill averages across the clan (e.g., `{ "autonomy": 2.9, "teamwork": 3.6 }`). Optional.
</ParamField>

**Example request:**

```json
{
  "clan": "Turing",
  "tl_id": 3,
  "total_coders": 8,
  "average_score": 72.4,
  "high_risk_count": 2,
  "top_struggling_topics": ["JOINs", "subqueries", "indexes"],
  "soft_skills_summary": {
    "autonomy": 2.9,
    "time_management": 3.4,
    "problem_solving": 3.1,
    "communication": 3.7,
    "teamwork": 3.6
  }
}
```

## How report generation works

The Node proxy (`POST /api/ia/generate-report`) passes the payload to the Python FastAPI service at `POST /generate-report` (`reports.py`).

<Steps>
  <Step title="Python service receives the payload">
    The `generate_report` endpoint in `reports.py` accepts a `ReportRequest` Pydantic model with the clan data fields.
  </Step>
  <Step title="Prompt is constructed">
    A detailed prompt is assembled with the clan's data and instructions for the LLM to produce exactly three sections: a current state summary, main risks and concerns, and 3–5 concrete action items.

    The LLM is instructed to return **only valid JSON** with no markdown fences.
  </Step>
  <Step title="Groq generates the report">
    The service calls `groq.chat.completions.create` with model `llama-3.3-70b-versatile` (configurable via the `MODEL_NAME` environment variable), temperature 0.6, and `max_tokens: 1000`.

    The system prompt explicitly constrains the output to valid JSON:
    ```
    "You are an educational analyst. Respond only with valid JSON, no markdown."
    ```
  </Step>
  <Step title="JSON is parsed and validated">
    If the model wraps the output in markdown code fences (`` ` ``), the service strips them before parsing. A `json.JSONDecodeError` returns HTTP 500 — the error is logged and the request fails rather than saving corrupt data.
  </Step>
  <Step title="Report is saved to ai_reports">
    A row is inserted into `ai_reports` with `target_type = 'clan'`, the TL's ID as `target_id`, the generated `summary_text`, `risk_level`, `recommendations`, the clan name as `clan_id`, and `viewed_by_tl = false`.
  </Step>
  <Step title="Generation is logged to ai_generation_log">
    A row is inserted into `ai_generation_log` with `agent_type = 'report_generator'`, the input payload, the full output JSON, the model name, and `success = true`.
  </Step>
</Steps>

## Report JSON structure

The LLM returns a JSON object that is stored in `ai_reports` and returned to the caller:

```json
{
  "report_title": "Clan Turing — Performance Report",
  "generated_date": "March 15, 2026",
  "risk_level": "medium",
  "summary": "Clan Turing shows moderate overall performance with an average score of 72.4/100...",
  "risks": "Two coders have active high-risk flags. Autonomy scores are below the 3.0 threshold...",
  "recommendations": "Schedule one-on-one sessions with at-risk coders. Assign guided activities for JOINs and subqueries...",
  "action_items": [
    "Review and update the learning plans of the two high-risk coders.",
    "Dedicate one session this week to subquery practice.",
    "Recognize top-performing coders to boost clan morale."
  ]
}
```

## Report risk level

The `risk_level` field in the report is one of `low`, `medium`, `high`, or `critical`, determined by the LLM from the data it receives. This value is stored in `ai_reports.risk_level` using the `risk_level_enum` type.

<Info>
  The report's `risk_level` is the LLM's holistic assessment of the clan's situation. It is independent of the per-coder `calculated_risk_level` in `v_coder_risk_analysis`, though both are informed by the same underlying data.
</Info>

## The `ai_reports` table

```sql
CREATE TABLE ai_reports (
  id           SERIAL PRIMARY KEY,
  target_type  report_target_enum NOT NULL,
  target_id    INT NOT NULL,
  summary_text TEXT NOT NULL,
  risk_level   risk_level_enum,
  recommendations TEXT,
  generated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  viewed_by_tl BOOLEAN DEFAULT FALSE
);
```

| Column | Type | Notes |
|---|---|---|
| `target_type` | `report_target_enum` | `coder`, `clan`, or `cohort`. |
| `target_id` | `INT` | For clan reports: the TL's user ID. |
| `summary_text` | `TEXT` | The LLM's current state summary paragraph. |
| `risk_level` | `risk_level_enum` | `low`, `medium`, `high`, or `critical`. |
| `recommendations` | `TEXT` | The LLM's recommendations paragraph. |
| `generated_at` | `TIMESTAMP` | Set automatically on insert. |
| `viewed_by_tl` | `BOOLEAN` | `false` until the TL opens the report. Defaults to `false`. |

The `viewed_by_tl` flag can be used to surface unread reports in the TL interface — reports where `viewed_by_tl = false` are treated as new.

Row-level security on `ai_reports` uses the `reports_tl_only` policy, which restricts all `SELECT` access to users with role `tl`.

## The `ai_generation_log` table

Every AI generation call — whether a learning plan, a report, or a risk detection run — is recorded in `ai_generation_log`. This table provides a full audit trail.

```sql
CREATE TABLE ai_generation_log (
  id                SERIAL PRIMARY KEY,
  coder_id          INT,
  agent_type        ai_agent_enum NOT NULL,
  input_payload     JSONB NOT NULL,
  output_payload    JSONB NOT NULL,
  model_name        VARCHAR(100),
  execution_time_ms INT,
  success           BOOLEAN DEFAULT TRUE,
  error_message     TEXT,
  generated_at      TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  FOREIGN KEY (coder_id) REFERENCES users(id) ON DELETE SET NULL
);
```

| Column | Type | Notes |
|---|---|---|
| `agent_type` | `ai_agent_enum` | `learning_plan`, `report_generator`, or `risk_detector`. For AI reports, this is always `report_generator`. |
| `input_payload` | `JSONB` | The data sent to the LLM (clan name, coder count, etc.). |
| `output_payload` | `JSONB` | The full JSON returned by the LLM. |
| `model_name` | `VARCHAR` | The model used (e.g., `llama-3.3-70b-versatile`). |
| `execution_time_ms` | `INT` | Wall-clock time for the LLM call in milliseconds. |
| `success` | `BOOLEAN` | `false` if the call failed or JSON parsing failed. |
| `error_message` | `TEXT` | Populated on failure. |

The `ai_log_tl_only` RLS policy restricts read access to users with role `tl`.

## PDF report — `GET /api/python/generate-pdf/:clan`

In addition to the AI-written report, the Python service can generate a structured PDF document for any clan. This endpoint does not use the LLM — it fetches coder data directly from Supabase and builds the PDF using `fpdf2`.

```bash
GET /generate-pdf/{clan}
```

<ParamField path="clan" type="string" required>
  The clan identifier (e.g., `Turing`).
</ParamField>

The PDF includes:
- A header with the clan name and generation date.
- An overview section: total coders, average score, and count of completed diagnostics.
- A per-coder profile section with email, Moodle average, current week, and all five soft skill scores and learning style.

The response streams as `application/pdf` with a `Content-Disposition: attachment` header, so browsers trigger a file download automatically.

<Warning>
  If no coders are found in the specified clan, the endpoint returns HTTP 404. Make sure the `clan` parameter exactly matches the clan identifiers stored in the `users.clan_id` column (`Turing`, `Tesla`, `McCarthy`).
</Warning>

In `dashboard.html`, the **PDF** button in the coder detail panel triggers a request to this endpoint to download the clan report.
