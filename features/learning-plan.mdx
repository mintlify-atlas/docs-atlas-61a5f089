---
title: "AI-Generated Learning Plans"
description: "How Kairo builds personalized 4-week study plans using soft skills, Moodle progress data, and the Groq LLM."
---

A learning plan is a 4-week, 20-day study program generated by the Groq LLM and tailored to a specific coder. Each plan contains daily activities structured around the coder's learning style and targets their weakest soft skill.

Plans are stored in the `complementary_plans` table and complement — not replace — the official Moodle curriculum.

## Plan structure

Every plan spans exactly **4 weeks × 5 days = 20 days**. Each day contains two activities:

- **Technical activity** (45 minutes) — a coding task, project, or deep-dive formatted for the coder's learning style.
- **Soft skill activity** (20 minutes) — a focused exercise targeting the coder's weakest soft skill dimension.

Day 20 (Week 4, Day 5) is always a **performance test simulation** — a timed, integrative exercise that mirrors the official Riwi performance evaluation.

## Plan types

<Tabs>
  <Tab title="Interpretive">
    Generated for coders who are new or have no Moodle progress data yet. The plan is based entirely on the soft skills assessment from onboarding.

    **Tone:** "Based on how you learn and your diagnostic results, here's how we'll work..."

    **When triggered:** Automatically after onboarding completes, as a fire-and-forget background request.

    **Context used by the LLM:**
    - Soft skills scores and learning style
    - Module name, description, and week structure
    - Topic list for the module
    - `current_week` (always `1` for new coders)
  </Tab>
  <Tab title="Analytical">
    Generated for returning coders who already have Moodle progress data. The plan incorporates actual academic performance to address detected weaknesses.

    **Tone:** "Last week we detected X — this week we address it with Y..."

    **When triggered:** By the coder from the dashboard via `POST /api/coder/plan/request` or by the Node proxy via `POST /api/ia/generate-plan`.

    **Additional context used by the LLM:**
    - Moodle `average_score`
    - `struggling_topics` array
    - `weeks_completed` history (last 3 weeks for trend analysis)
    - `current_week` in the module
  </Tab>
</Tabs>

## Plan generation flow

<Steps>
  <Step title="Request arrives at Node — POST /api/ia/generate-plan">
    The Node.js server receives the request. It queries `moodle_progress` for the coder's current module, week, and struggling topics. It resolves topic IDs to names from the `topics` table, then forwards a slim payload to the Python service.
  </Step>
  <Step title="Node proxies to Python — POST /generate-plan">
    The Node server calls the Python FastAPI service with a JSON payload containing `coder_id`, `module_id`, `plan_type`, `current_week`, and `struggling_topics`.
  </Step>
  <Step title="Python fetches context from Supabase">
    The Python service (`roadmap.py`) fetches all required context directly from Supabase:
    - Soft skills assessment (`soft_skills_assessment`)
    - Module details (`modules`)
    - Week structure (`weeks`)
    - Topic list (`topics`)
    - For `analytical` plans: auto-fetches the latest `moodle_progress` row if not provided

    If no soft skills assessment is found, the request fails with HTTP 400 — the coder must complete onboarding first.
  </Step>
  <Step title="Prompt builder constructs the LLM context">
    `prompt_builder.py` builds a detailed prompt that includes:
    - The coder's learning style and per-skill scores with the weakest skill highlighted
    - Style-specific activity format instructions (e.g., kinesthetic learners get hands-on code challenges from minute one)
    - Adaptation rules based on low scores (e.g., step-by-step instructions for autonomy < 3)
    - A verified resource catalog with real URLs per module and style
    - For analytical plans: score analysis, trend detection, and a recovery strategy
  </Step>
  <Step title="Groq LLM generates the plan JSON">
    The service calls `groq.chat.completions.create` with model `llama-3.3-70b-versatile` (configurable via `MODEL_NAME`), temperature 0.7, and `max_tokens: 8192`. The response format is set to `json_object` to guarantee parseable output.

    If the Groq call fails or returns unparseable JSON, a safe fallback plan structure is returned so the database insert never crashes.
  </Step>
  <Step title="Previous active plan is deactivated">
    Before saving the new plan, `db_manager.deactivate_plans()` sets `is_active = false` on all existing active plans for that coder. Only one plan is active at a time.
  </Step>
  <Step title="Plan is saved to complementary_plans">
    The full plan JSON is stored in `plan_content`. Snapshots of the soft skills and Moodle status at generation time are stored in `soft_skills_snapshot` and `moodle_status_snapshot`.
  </Step>
  <Step title="Activities are written to plan_activities">
    `db_manager.save_plan_activities()` iterates the plan's weeks and days and inserts up to two rows per day (one `technical`, one `soft_skill`) into `plan_activities`. This populates the table with live data the TL can query.
  </Step>
  <Step title="Generation is logged and response returned">
    The generation event is recorded in `ai_generation_log`. The Python service returns the plan JSON, `plan_id`, and metadata. Node relays this to the original caller.
  </Step>
</Steps>

## Triggering plan generation

<Tabs>
  <Tab title="Automatic (after onboarding)">
    Plan generation triggers automatically in the background after a coder submits their diagnostic answers. The `diagnosticControllers.js` fires a fetch to the Python service and responds to the frontend immediately without waiting.

    ```javascript
    // diagnosticControllers.js — fire-and-forget after saving the assessment
    _triggerInterpretivePlan(req.session.userId, moduleId, currentWeek);
    ```
  </Tab>
  <Tab title="Manual (from dashboard)">
    A coder can request a new plan from their dashboard. The `POST /api/coder/plan/request` endpoint responds immediately and fires the Python call asynchronously.

    ```bash
    POST /api/coder/plan/request
    ```

    ```json
    {
      "plan_type": "analytical",
      "current_week": 3
    }
    ```

    **Response:**
    ```json
    {
      "requested": true,
      "message": "Plan en generación. Reintenta en unos segundos."
    }
    ```

    The dashboard polls `GET /api/coder/plan` after a short delay to check if the new plan is ready.
  </Tab>
  <Tab title="Via Node proxy">
    Authenticated coders can call the Node proxy directly:

    ```bash
    POST /api/ia/generate-plan
    ```

    This is a synchronous call — Node awaits the Python response before returning.

    ```json
    {
      "topic": "Módulo 2 — Semana 3",
      "additionalTopics": ["JOINs", "subqueries"]
    }
    ```

    The `topic` and `additionalTopics` fields are optional. Module and week are resolved automatically from the coder's `moodle_progress`.
  </Tab>
</Tabs>

## API reference

### Get the active plan

```
GET /api/coder/plan
```

Returns the coder's currently active plan with progress state.

**Example response**

```json
{
  "hasPlan": true,
  "plan": {
    "id": 17,
    "moduleId": 4,
    "planContent": { "plan_type": "analytical", "weeks": [] },
    "targetedSoftSkill": "problem_solving",
    "generatedAt": "2026-03-15T09:00:00.000Z",
    "completedDays": { "1": { "completedAt": "2026-03-15T11:00:00.000Z" } },
    "currentDay": 2,
    "completedCount": 1,
    "isComplete": false
  }
}
```

### Get plan details

```
GET /api/coder/plans/:planId
```

Returns a plan together with all its `plan_activities` rows.

### Complete a day

```
POST /api/coder/plan/:planId/day/:day/complete
```

Marks a plan day as complete. Valid day values are 1–20. Awards Kairo score points for each completed day, with a bonus on day 20.

**Example response**

```json
{
  "success": true,
  "completedDays": { "1": { "completedAt": "..." }, "2": { "completedAt": "..." } },
  "completedCount": 2,
  "nextDay": 3,
  "isComplete": false,
  "kairoScore": 55
}
```

### Generate a plan (Node proxy)

```
POST /api/ia/generate-plan
```

Requires role `coder`. Synchronous — waits for the Python service to respond.

<ParamField body="topic" type="string">
  Human-readable label for the plan focus (e.g., `"Módulo 2 — Semana 3"`). Defaults to `"Módulo {n} — Semana {n}"` if omitted.
</ParamField>

<ParamField body="additionalTopics" type="array">
  Extra topic names to include in the plan context. Optional.
</ParamField>

## Database tables

### `complementary_plans`

```sql
CREATE TABLE complementary_plans (
  id                     SERIAL PRIMARY KEY,
  coder_id               INT NOT NULL,
  module_id              INT NOT NULL,
  plan_content           JSONB NOT NULL,
  priority_level         priority_level_enum DEFAULT 'medium',
  soft_skills_snapshot   JSONB,
  moodle_status_snapshot JSONB,
  is_active              BOOLEAN DEFAULT TRUE,
  generated_at           TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  FOREIGN KEY (coder_id)  REFERENCES users(id) ON DELETE CASCADE,
  FOREIGN KEY (module_id) REFERENCES modules(id) ON DELETE CASCADE
);
```

| Column | Type | Notes |
|---|---|---|
| `plan_content` | `JSONB` | Full LLM-generated plan with weeks, days, and activities |
| `priority_level` | `priority_level_enum` | `low`, `medium`, or `high` |
| `soft_skills_snapshot` | `JSONB` | Copy of the coder's soft skills at generation time |
| `moodle_status_snapshot` | `JSONB` | Moodle context used to generate the plan (includes `plan_type`) |
| `is_active` | `BOOLEAN` | Only one plan per coder is active at a time |
| `targeted_soft_skill` | `VARCHAR` | The weakest skill dimension the plan targets |

### `plan_activities`

```sql
CREATE TABLE plan_activities (
  id                     SERIAL PRIMARY KEY,
  plan_id                INT NOT NULL,
  day_number             INT NOT NULL,
  title                  VARCHAR(200) NOT NULL,
  description            TEXT,
  estimated_time_minutes INT,
  activity_type          activity_type_enum,
  skill_focus            VARCHAR(100),
  FOREIGN KEY (plan_id) REFERENCES complementary_plans(id) ON DELETE CASCADE
);
```

| Column | Type | Notes |
|---|---|---|
| `day_number` | `INT` | 1–20, matching the plan's day structure |
| `activity_type` | `activity_type_enum` | `guided`, `semi_guided`, or `autonomous` |
| `skill_focus` | `VARCHAR` | The week's technical focus topic or the targeted soft skill |

<Note>
  Up to 40 rows are inserted per plan (2 per day × 20 days). If the Python service cannot parse activities from the generated JSON, the insert is skipped without failing the plan save.
</Note>
